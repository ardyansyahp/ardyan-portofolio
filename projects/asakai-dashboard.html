03. Case Study: Analytics & Control

Asakai Dashboard & DataQC

Klien: PT. Mada Wikri Tunggal
Peran: Lead Developer
Stack: Laravel 12, Livewire 3, Alpine.js, Tailwind CSS
Efisiensi: 7.5 Jam Dihemat / Minggu

Masalah Operasional Awal

Setiap pagi jajaran manajemen mengadakan rapat koordinasi cepat (Morning Briefing / Asakai) untuk mengevaluasi parameter operasional dari hari sebelumnya. Sebelum dashboard Asakai dideploy:

  • Persiapan Laporan Manual: Setiap kepala departemen (Produksi, PPIC, QC, Moldshop, Maintenance, HSE, & GA) harus meluangkan waktu 1–2 jam setiap sore atau pagi hari untuk mengumpulkan data dari lembaran kertas dan mencatatnya ke file Excel masing-masing.
  • Keterlambatan Informasi: Data sering kali baru selesai direkap siang hari, padahal rapat dilakukan jam 8 pagi. Rapat sering kali berjalan menggunakan data lama atau estimasi verbal yang tidak akurat.
  • Kesulitan Visualisasi: Laporan disajikan dalam bentuk spreadsheet rumit yang sulit dibaca dari jarak jauh pada layar monitor TV ruang rapat.

Pendekatan Arsitektur Teknologi yang Dipilih

Asakai Dashboard dirancang sebagai single source of truth yang menyatukan data operasional dari 7 departemen berbeda. Sistem ini diletakkan pada TV besar di dinding pabrik dan ruang manajemen, teroptimasi untuk konsumsi visual kepadatan data tinggi (High Data Density).

Tantangan Teknis & Rekayasa Performa Database

Tantangan rekayasa perangkat lunak terbesar dalam merancang dashboard telemetri real-time ini adalah keterbatasan performa database. Melakukan operasi agregasi (seperti SUM, COUNT, dan AVERAGE) secara real-time pada 7 tabel departemen yang memiliki jutaan baris data transaksi harian dapat mengunci tabel (*table lock*) dan memperlambat kueri pada lini produksi utama ERP.

Solusi Rekayasa Performa Database:

  • Mekanisme Caching Cerdas: Penerapan caching data agregasi menggunakan Laravel Cache dengan tag berdurasi 5 menit. Data laporan tidak ditarik langsung dari database pada setiap request, melainkan dibaca dari cache memori yang cepat.
  • Tampilan Teragregasi (Materialized Aggregates): Membuat tabel agregasi khusus yang bertindak sebagai *Materialized Views*. Script cronjob di latar belakang berjalan secara berkala untuk melakukan kalkulasi data historis yang sudah tetap (data bulan-bulan sebelumnya) dan menyimpannya di tabel agregasi, sehingga query dashboard hanya memproses data hari berjalan saja.

Berikut adalah cuplikan kode implementasi caching agregasi data di stasiun TV Asakai untuk meminimalkan beban query database: 

// App/Http/Controllers/DashboardController.php
namespace App\Http\Controllers;

use Illuminate\Support\Facades\Cache;
use Carbon\Carbon;
use App\Models\HseAccident;
use App\Models\ProduksiOee;

class DashboardController extends Controller
{
    public function index()
    {
        $currentYear = Carbon::now()->year;

        // Membungkus query agregasi berat di dalam cache berdurasi 300 detik (5 menit)
        $data = Cache::remember('asakai_metrics_' . $currentYear, 300, function() use ($currentYear) {
            return [
                // Menghitung kecelakaan HSE dengan kueri teragregasi
                'hse_accident' => HseAccident::whereYear('date', $currentYear)->sum('frequency') ?? 0,
                // Mengambil OEE rata-rata stasiun kerja
                'oee_average' => ProduksiOee::whereYear('date', $currentYear)->avg('value') ?? 0
            ];
        });

        return view('asakai.dashboard', compact('data'));
    }
}

Hasil Akhir yang Terukur

Dashboard Asakai berhasil mengeliminasi proses manual penyusunan laporan harian yang memakan waktu 1 hingga 2 jam per hari bagi jajaran manajemen lintas 7 departemen. Penghematan waktu operasional ini memberikan peningkatan efisiensi yang signifikan pada alur kerja persiapan rapat pagi:

[ FORMULA PENGHEMATAN WAKTU KOORDINASI ]
Waktu Dihemat = 1.5 Jam / Hari × 5 Hari Kerja / Minggu = 7.5 Jam / Minggu

Dampak Terukur Hasil Penerapan:

  • Waktu Persiapan Rapat: Berkurang dari 1.5 jam menjadi 0 menit (semua parameter ter-update otomatis secara real-time).
  • Kecepatan Respon Masalah (Lead Time): Turun hingga 40% karena deviasi parameter target (seperti defect OEE) langsung terdeteksi jam 8 pagi dan bisa diselesaikan pada hari yang sama.
  • Penyebaran Informasi Lintas Departemen: Koordinasi antar departemen berjalan transparan tanpa hambatan data (*data silos*).
