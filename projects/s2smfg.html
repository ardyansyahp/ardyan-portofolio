Masalah Operasional Awal

Sebelum sistem S2SMFG dideploy, seluruh alur kerja pelacakan material dan produksi di pabrik PT. Mada Wikri Tunggal masih berbasis kertas manual. Lembar "Kartu Travel" digunakan untuk mencatat perpindahan barang jadi dari mesin cetak ke area karantina QC, lalu ke gudang barang jadi.

Metode manual ini menyebabkan beberapa masalah kritis pada rantai pasok:

  • Kehilangan Data Fisik: Kartu travel kertas sering kotor, robek, atau bahkan hilang di area lantai pabrik yang sibuk, sehingga melumpuhkan pelacakan riwayat lot produksi.
  • Delay Sinkronisasi Inventaris: Stok barang di gudang hanya direkap sekali dalam 24 jam ke dalam spreadsheet Excel. PPIC tidak memiliki data riil saat melakukan perencanaan jadwal produksi harian.
  • Human Error Tinggi: Kesalahan pengetikan kode part yang mirip menyebabkan selisih inventaris bernilai puluhan juta rupiah.

Teknik Optimasi Indeks Database:

  • Composite Unique Key: Kolom qr_hash dikonfigurasi sebagai Unique Key untuk mencegah box finished good divalidasi lebih dari satu kali (mencegah duplikasi data).
  • Foreign Key Indexing: Indeks dibuat secara eksplisit pada kolom foreign key spk_id dan part_id di tabel transaksi t_finish_good_out untuk mengeliminasi operasi full table scan saat proses join query laporan bulanan.
  • Covering Index: Membuat indeks komposit idx_spk_status (spk_id, status) untuk mempercepat laju kalkulasi dashboard real-time yang hanya mencari box bertanda 'scanned' dari SPK aktif tertentu.

Implementasi Logika Bisnis (Backend Controllers)

Tantangan teknis terbesar adalah menghindari anomali data serta mempertahankan kecepatan transaksi melintasi puluhan stasiun kerja secara simultan. Berikut adalah bagaimana 3 titik krusial dalam alur produksi ditangani oleh backend Laravel (didukung oleh penerapan sertifikasi DBA):

1. Fase Awal: Penerimaan Material (Receiving)

Mencegah selisih gudang sejak barang datang. Sistem otomatis memverifikasi penerimaan dengan Purchase Order (PO) dan menambah stok bahan baku secara real-time menggunakan Database Transaction.

// App/Http/Controllers/Api/ReceivingController.php
public function processReceiving(Request $request) {
    return DB::transaction(function() use ($request) {
        // 1. Lock Purchase Order untuk mencegah penerimaan ganda (race condition)
        $po = PurchaseOrder::where('po_number', $request->po_number)->lockForUpdate()->firstOrFail();
        
        // 2. Validasi Qty Kedatangan vs Sisa Qty Order
        if ($request->qty_received > $po->remaining_qty) {
            return response()->json(['error' => 'Over receipt detected!'], 422);
        }

        // 3. Update Master Inventory Material
        Inventory::incrementMaterialStock($po->material_id, $request->qty_received);
        $po->decrement('remaining_qty', $request->qty_received);
        
        return response()->json(['success' => 'Material diterima & stok bertambah.']);
    });
}

2. Fase Tengah: Produksi & Backflushing (WIP)

Saat mesin mencetak produk, operator hanya melaporkan "Hasil Jadi". Sistem menggunakan Bill of Materials (BOM) untuk memotong stok bijih plastik secara otomatis (metode backflush), meminimalisir input manual. 

// App/Http/Controllers/Api/ProductionController.php
public function recordProductionYield(Request $request) {
    return DB::transaction(function() use ($request) {
        // 1. Catat Barang Setengah Jadi (WIP) / Hasil Produksi
        $yield = ProductionYield::create($request->all());
        
        // 2. Ambil resep material (BOM) untuk produk ini
        $bom = BillOfMaterial::getRecipeFor($request->part_id);
        
        // 3. Auto-deduct (Backflush) stok material mentah (BOM x Qty Produksi)
        Inventory::decrementMaterialStock($bom->raw_material_id, ($bom->usage_per_pcs * $yield->qty));
        
        return response()->json(['success' => 'Hasil dicatat, stok mentah terpotong otomatis.']);
    });
}

3. Fase Akhir: Pengiriman Finish Good (Shipping)

Mencegah double-scan box barang jadi ke dalam truk. Menerapkan Optimistic Locking (lockForUpdate) dan komputasi MD5 hash pada constraint database untuk akurasi pengiriman 100%.

// App/Http/Controllers/Api/FinishGoodScanController.php
public function processScan(Request $request) {
    return DB::transaction(function() use ($request) {
        // 1. Lock Row SPK (Surat Perintah Kerja) untuk mencegah race condition
        $spk = TSpk::where('id', $request->spk_id)->lockForUpdate()->firstOrFail();

        // 2. Periksa duplikasi scan box melalui Unique Constraint Hash (MD5)
        if (TFinishGoodOut::where('qr_hash', md5($request->qr_content))->exists()) {
            return response()->json(['error' => 'Box ini sudah discan sebelumnya'], 422);
        }

        // 3. Catat scan pengiriman barang & update progress SPK
        TFinishGoodOut::recordScan($request);
        $spk->increment('total_scanned_qty', $request->qty);

        return response()->json(['success' => 'Box berhasil divalidasi ke truk pengiriman.']);
    });
}

Konteks Infrastruktur & Rationale Deployment cPanel

PT. Mada Wikri Tunggal merupakan perusahaan manufaktur konvensional yang secara bisnis sudah matang tetapi secara kultur teknologi digital masih berkembang. Selama proses rekayasa sistem ERP ini, kami dihadapkan pada kendala infrastruktur nyata:

  • Keterbatasan Sumber Daya Manusia: Tim IT sangat minimalis (kurang orang), sehingga pengerjaan seluruh siklus pengembangan (SDLC) dilakukan secara mandiri.
  • Infrastruktur Jaringan & Server Pabrik: Belum tersedianya infrastruktur server lokal (on-premise) yang memadai untuk penanganan skalabilitas mandiri, serta tingkat pemahaman digitalisasi dan kesadaran implementasi sistem yang masih bertahap di lapangan.

Mengapa Menggunakan Shared Hosting cPanel?
Mengingat kendala-kendala di atas, keputusan untuk men-deploy aplikasi ERP produksi utama pada shared hosting cPanel merupakan keputusan rekayasa pragmatis. Deployment di cPanel sangat meminimalkan beban pengelolaan server harian (server administration/maintenance overhead) sehingga pengembang tunggal dapat berfokus penuh pada stabilitas kode dan logika bisnis rantai pasok tanpa tersita oleh urusan sysadmin dan pemeliharaan server secara manual.

[ Rekomendasi Jamstack ]
Meskipun sistem operasional pabrik berjalan di shared hosting cPanel karena tuntutan legacy, untuk situs web portofolio statis/profil profesional sangat direkomendasikan menerapkan arsitektur Jamstack (seperti Netlify atau Vercel). Jamstack menawarkan kecepatan pemuatan maksimal melalui CDN global, tingkat keamanan tinggi tanpa database, serta biaya operasional gratis (nihil) untuk pengembang.

Hasil Akhir yang Terukur

Implementasi ERP S2SMFG berhasil mengubah total operasi lantai pabrik. Data transaksi terkonsolidasi secara waktu nyata. Laju beban kerja operasional yang berhasil ditangani sistem dapat dimodelkan secara matematis untuk membuktikan kapasitas beban kerjanya:

[ PEMODELAN BEBAN TRANSAKSI HARIAN ]
Beban Transaksi Harian = 100 SPK + 500 Transaksi Inventaris = 600 Aktivitas / Hari

Dengan asumsi waktu operasional pabrik berjalan selama 8 jam per hari, maka laju pemrosesan transaksi rata-rata yang dikelola sistem adalah:
Laju Pemrosesan = 600 Transaksi / 8 Jam = 75 Transaksi / Jam

Dampak Terukur Hasil Penerapan:

  • Reduksi Kehilangan Dokumen: Kehilangan lembar SPK & Kartu Travel fisik turun hingga 0% karena seluruh alur penugasan dipantau di layar stasiun kerja.
  • Akurasi Rekonsiliasi Inventaris: Laporan perbedaan stok di PPIC berhasil diminimalisir hingga mendekati 99.9% kecocokan fisik berkat validasi QR code unik.
  • Kecepatan Perencanaan: Tim PPIC dapat merancang jadwal produksi 2 jam lebih cepat setiap sore karena data stok bahan baku ter-update secara instan tanpa rekap manual.
